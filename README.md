# Тестовое задание 1С — внешняя обработка «ОбработкаВедомостей»

Платформа **8.3.25**, управляемые формы. Внешняя обработка с двумя командами:
**«Заполнить нужное»** и **«Удалить лишнее»**.

Файлы:

| Файл | Что это |
|------|---------|
| `ОбработкаВедомостей.Форма.МодульФормы.bsl` | Модуль основной формы (вся логика) |
| `ОбработкаВедомостей.ФормаСообщения.МодульФормы.bsl` | Модуль вспомогательной формы вывода результата |
| `check_logic.py` | Автопроверка алгоритма на данных задания (не часть обработки) |
| `README.md` | Этот файл: разбор, сборка в Конфигураторе, обоснование вариантов |

Готового `.epf` в комплекте нет намеренно — форму быстрее и надёжнее собрать
в Конфигураторе (5 минут по инструкции ниже), а весь смысловой код уже готов.

---

## 1. Что делает обработка

**«Заполнить нужное»** (сервер, один пакетный запрос):
1. Программно формирует таблицу ведомостей, у которых **дата начала действия
   первого активного договора** в табличной части `равна`/`не равна`
   `минимальной`/`максимальной` дате начала действия договора в этой ТЧ.
   Ведомости без активного договора в отбор не попадают.
2. Выводит таблицу на форму, видимая колонка — только `Ведомость`.
3. Сортирует по возрастанию даты документа (в запросе).
4. Светло-зелёным подсвечивает строки, где **последний** статус ведомости
   (по регистру `ИсторияСтатусовДокументов`) = `Акцептован`.

**«Удалить лишнее»** (клиент, без обращения к серверу):
1. Удаляет ведомости с **чётным** количеством договоров в ТЧ.
2. Выводит номера удалённых документов (сортировка по возрастанию номера):
   `Удалено документов: 3 (БТ02-000256, БТ02-000512, ЛР04-000128).`

Сноски 1–2 задания («равна/не равна», «минимальной/максимальной») —
вариативные, поэтому вынесены на форму двумя переключателями и покрывают все
4 комбинации.

---

## 2. Замеченная опечатка в задании

В «Примере 1» напечатано:

> Удалено документов: 4 (БТ02-000016, БТ02-000032, **БТ02-000256**, ЛР04-001024).

Документа `БТ02-000256` в данных нет. В таблице до удаления есть `ЛР04-000256`
(8 договоров — чётное, удаляется). Правильный вывод:

> Удалено документов: 4 (БТ02-000016, БТ02-000032, **ЛР04-000256**, ЛР04-001024).

`check_logic.py` воспроизводит оба примера задания и подтверждает это (`python check_logic.py` → `OK`).

## 3. Индивидуальные варианты дополнительного функционала

Раздел «Дополнительный функционал» зависит от данных кандидата. Здесь —
Камиль Гильфанов; допущения ниже. **Если что-то из этого не так — правки
точечные:**

| Пункт | Условие | Наш случай | Реализация | Как поменять |
|-------|---------|-----------|------------|--------------|
| (1) Исключение при заполнении | дата отправки решения | **04.07 → диапазон 1–10** | исключаем ведомости, где `День(даты) = День(ТекущаяДата())` | в `ЗаполнитьТаблицуВедомостейНаСервере` заменить условие: 11–20 → `День(дата) % 2 = 0`; 21–31 → `Месяц(дата) % 2 = 1` (альтернативы уже указаны в комментарии) |
| (2) Условное оформление | пол | **мужской** | **программно** (`НастроитьУсловноеОформление`) | женский → убрать вызов и настроить оформление **интерактивно** конструктором формы на тот же отбор |
| (3) Вывод информации | длина фамилии | **«Гильфанов» = 9 → диапазон 8–9** | **создать и открыть произвольную форму** (`ФормаСообщения`) | см. таблицу задания: `<6` → `Сообщить()`; `6–7` → `СообщениеПользователю`; `10–11` → элемент осн. формы; `12–13` → вызов исключения; `>13` → `Предупреждение` |

> Важно: из-за пункта (1) вывод «Заполнить нужное» **намеренно** отличается от
> «Примера 1/2» задания в те дни, когда день даты ведомости совпадает с днём
> запуска — это и есть индивидуальное условие. Базовое поведение (без пункта 1)
> совпадает с примерами один-в-один (проверено `check_logic.py`).

## 4. Объекты, которые нужно создать в тестовой базе

`cf/dt` не предоставлены — окружение воссоздаётся самостоятельно. Имена должны
**точно** совпадать с приведёнными (на них ссылается код).

- **Перечисление** `СтатусыДокументов`: `Создан`, `Внутренний`, `Акцептован`, `Заблокирован`.
- **Справочник** `Договоры`: `Наименование` (строка, 100), `Активен` (булево),
  `НачалоДействия` (дата, состав — только дата).
- **Документ** `Ведомость`, ТЧ `Договоры` с реквизитом `Договор`
  (`СправочникСсылка.Договоры`).
- **Регистр сведений** `ИсторияСтатусовДокументов`, независимый непериодический,
  измерения: `ДатаЗаписи` (дата+время), `Документ` (составной, включает
  `ДокументСсылка.Ведомость`), `Статус` (`ПеречислениеСсылка.СтатусыДокументов`).

Данные из раздела «Пример данных» задания удобно завести как есть для проверки.

## 5. Сборка обработки в Конфигураторе

**Внешняя обработка** `ОбработкаВедомостей` (имя произвольно — имена форм код
получает в рантайме). Две формы: `Форма` (основная) и `ФормаСообщения`.

### Основная форма `Форма`

**Реквизиты:**
- `ТаблицаВедомостей` — `ТаблицаЗначений`, колонки:
  - `Ведомость` — `ДокументСсылка.Ведомость`
  - `Номер` — `Строка` (20)
  - `ДатаВедомости` — `Дата` (дата+время)
  - `КоличествоДоговоров` — `Число` (10)
  - `ПоследнийСтатус` — `ПеречислениеСсылка.СтатусыДокументов`
- `ВариантСравнения` — `Число` (1), список выбора: `0 → «равна»`, `1 → «не равна»`
- `ВариантЭкстремума` — `Число` (1), список выбора: `0 → «минимальной»`, `1 → «максимальной»`

**Команды:** `ЗаполнитьНужное`, `УдалитьЛишнее` (действия — одноимённые процедуры модуля).

**Элементы формы:**
- Командная панель с кнопками команд `ЗаполнитьНужное` и `УдалитьЛишнее`.
- Поле `ВариантСравнения` — вид **Переключатель** (радиокнопки «равна»/«не равна»).
- Поле `ВариантЭкстремума` — вид **Переключатель** («минимальной»/«максимальной»).
- Таблица на реквизите `ТаблицаВедомостей`: у колонок `Номер`, `ДатаВедомости`,
  `КоличествоДоговоров`, `ПоследнийСтатус` снять **Видимость** (остаётся только `Ведомость`).

Условное оформление конструктором **не** настраиваем — оно создаётся из кода
(`НастроитьУсловноеОформление`, т.к. пол мужской).

Вставить код из `ОбработкаВедомостей.Форма.МодульФормы.bsl` в модуль формы.

### Форма `ФормаСообщения`

- **Параметр:** `Текст` — `Строка`.
- **Реквизит:** `ТекстСообщения` — `Строка` (неограниченная).
- **Элемент:** поле на `ТекстСообщения` — многострочное, только просмотр, растянуть.
- Заголовок формы — например «Результат обработки».
- Вставить код из `ОбработкаВедомостей.ФормаСообщения.МодульФормы.bsl`.

## 6. Заметки по реализации (для ревьюера)

- **Клиент/сервер.** «Заполнить нужное» — один серверный вызов с пакетным
  запросом (агрегации и соединения на стороне СУБД). «Удалить лишнее» —
  полностью на клиенте: `Номер` и `КоличествоДоговоров` заранее положены в
  таблицу формы, поэтому лишних обращений к серверу и «ленивых» чтений ссылок нет.
- **Первый активный договор** = строка ТЧ с минимальным `НомерСтроки` среди строк
  с `Договор.Активен = Истина`; min/max берётся по датам **всех** договоров ТЧ.
- **Последний статус** = статус записи регистра с максимальной `ДатаЗаписи` по
  документу; `ВЫРАЗИТЬ(… КАК Документ.Ведомость)` отсекает прочие типы составного
  измерения; `МАКСИМУМ(Статус)` — детерминированный tie-break (в данных задания
  совпадений `ДатаЗаписи` нет).
- **Отбор по переключателям** сделан в коде, а не в запросе: результат — одна
  строка на ведомость, набор мал, и это избавляет от ветвления текста запроса под
  4 комбинации. Если объём ведомостей станет большим — сравнение
  `ДатаПервогоАктивного` с min/max дешевле перенести в `ГДЕ` финального запроса
  через параметры.
- **Цвет** `Новый Цвет(221, 235, 199)` подобран под примеры; вынесен одной строкой.
- **Имя вспомогательной формы** собирается из `ЭтаФорма.ИмяФормы` — не зависит от
  имени, под которым обработку подключат. Если платформа вернёт неожиданное имя,
  строку можно захардкодить.
